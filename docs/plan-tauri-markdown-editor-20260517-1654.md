&gt; **NOTE**: user annotation here

## Approach

Create a new Linux-first Tauri v2 desktop app in the repository root using Vite, React, and TypeScript. Use Milkdown as the WYSIWYG Markdown editor because the requirement is Markdown-first WYSIWYG editing, and Milkdown is explicitly built for that use case. Keep the first version intentionally narrow: local files only, single document at a time, manual save plus dirty-state prompts, command-line opening for one Markdown file, no autosave, no recent-file persistence, no cloud sync, no collaborative editing, no auto-updater.

Chosen stack:

- Frontend: Vite + React + TypeScript in `src/`.
- Desktop shell: Tauri v2 in `src-tauri/`.
- Editor: Milkdown React integration in `src/features/editor/`.
- File access: Tauri dialog plugin for open/save selection, Tauri fs plugin for reading/writing selected paths, and Tauri CLI plugin for startup file arguments.
- State: local React state only; do not persist recent files or autosave recovery snapshots in v1.
- Styling: plain CSS modules or a small global CSS file; avoid a heavy UI framework for the first version.

Why this approach:

- Tauri keeps the app standalone and small compared to an Electron baseline.
- React/TypeScript is a conservative choice because Tauri officially supports a React template and Milkdown has React integration.
- Milkdown reduces the risk of hand-rolling Markdown editing behavior and makes Markdown serialization part of the editor model.
- Exact Markdown preservation is more important than WYSIWYG convenience for plan-cycle files. If a document contains `> **NOTE**:`, source mode must become the default and authoritative save view for that file. WYSIWYG remains available only as an explicit preview/editing toggle and must not be allowed to overwrite source text unless serializer tests prove exact note preservation.
- A single-document workflow keeps the first app fast and understandable.

Exit clauses:

- If Milkdown cannot preserve plan-cycle constructs (`> **NOTE**:`, fenced code blocks, checklists, headings) across open-save tests with `0` serializer exceptions for note lines, switch plan-cycle files to mandatory source editing with live preview using CodeMirror 6 and markdown-it; keep WYSIWYG only for files without plan-cycle notes.
- If Tauri JS `plugin-fs` absolute-path access via dialog-selected paths or CLI-provided paths is blocked during Linux manual QA, add Rust commands in `src-tauri/src/lib.rs` for `read_markdown_file`, `write_markdown_file`, and `stat_markdown_file`.
- If React + Milkdown bundle cold start exceeds `2 seconds` on the development Linux machine after production build, evaluate a lighter Milkdown setup first; if still above `2 seconds`, switch to CodeMirror 6 source editor plus preview for v1.

Target service API:

  markdown: string

Rules:

- Accept extensions `.md`, `.markdown`, and `.mdown`.
- Reject extensions outside that set before reading.
- Check file metadata before reading; reject above `10 * 1024 * 1024` bytes.
- Normalize displayed file name from the final path segment only; do not parse platform-specific full paths in UI components.
- Return typed failures to the UI instead of throwing raw plugin errors across component boundaries.
- Preserve UTF-8 text; if decoding fails, show `read-failed` and leave the current editor state unchanged.
- Detect line endings on load: if the content contains at least one `\r\n`, set `lineEnding` to `crlf`; otherwise set it to `lf`.
- When saving an existing document, normalize outgoing Markdown to `document.lineEnding`.
- For Save As, use the provided `options.lineEnding`; if absent, use `LF`.
- Detect plan-cycle files with `/^> \*\*NOTE\*\*:/m` and set `hasPlanCycleNotes`.
- Recompute `byteLength` with `new TextEncoder().encode(markdown).byteLength` after line-ending normalization.

Success criteria:

- Unit tests cover extension validation, size limit handling, cancellation, line-ending detection/preservation, plan-cycle note detection, byte length recomputation, and failure mapping.
- Opening a cancelled dialog does not change the active document.
- Saving a document updates `savedAt` and clears dirty state in the app store.

## Edge Cases and Risks

- **Risk**: WYSIWYG serialization changes Markdown formatting that plan-cycle relies on.
  - **Likelihood**: medium.
  - **Impact**: annotations may stop matching `> **NOTE**:` and plan-cycle review cannot process them.
  - **Mitigation**: add fixtures and regex tests for plan-cycle notes; open plan-cycle files in source mode by default; manually grep saved files.
  - **Exit clause**: if `> **NOTE**:` cannot be preserved exactly in `2` representative fixtures through WYSIWYG, make WYSIWYG unavailable for files containing plan-cycle notes and save only from source mode.

- **Risk**: Tauri fs, dialog, or CLI permission names differ from the planned capability file.
  - **Likelihood**: medium.
  - **Impact**: app launches but cannot read selected files, save selected files, or parse startup arguments.
  - **Mitigation**: use generated Tauri schema/build errors to choose narrow permissions; rely on dialog-selected path scoping first; verify `cli:default` for argument reads.
  - **Exit clause**: if JS fs plugin access fails after dialog selection or CLI startup on Linux manual QA, implement Rust file commands with explicit path validation.

- **Risk**: Large Markdown files cause editor stalls.
  - **Likelihood**: medium.
  - **Impact**: typing or mode switching becomes visibly slow.
  - **Mitigation**: reject above `10 MB`; warn above `2 MB`; throttle editor change events to one per animation frame.
  - **Exit clause**: if `50,000` characters exceed `50 ms p95` typing latency in production build, disable WYSIWYG for files above `50,000` characters and force source mode with preview.

- **Risk**: Unsaved edits are lost during open/new/close flows.
  - **Likelihood**: medium.
  - **Impact**: user data loss.
  - **Mitigation**: centralize commands in `src/app/commands.ts`; every destructive transition checks `dirty`; failed save keeps dirty state.
  - **Exit clause**: if close interception is unreliable on Linux, keep internal Open/New protections, add a persistent status warning about OS-close protection, and document the limitation; do not add autosave in v1.

- **Risk**: Linux packaging prerequisites differ across distributions.
  - **Likelihood**: medium.
  - **Impact**: the app may build on the current Linux machine but fail on another Linux distribution.
  - **Mitigation**: document official Tauri Linux prerequisites in `README.md`; verify development and production build on the current machine first.
  - **Exit clause**: if distribution across multiple Linux distros is required immediately, add a Linux CI build before feature work continues.

- **Risk**: CLI-provided absolute paths are not automatically included in Tauri fs scope.
  - **Likelihood**: medium.
  - **Impact**: dialog opening works, but `amorist path/to/file.md` cannot read the file.
  - **Mitigation**: test CLI startup separately from dialog opening; keep validation shared across both paths.
  - **Exit clause**: if CLI paths cannot be read through JS fs permissions, implement Rust commands for CLI path reads/writes and keep dialog paths on plugin-fs if they work.

## Failure Modes and Degradation

- **File open dialog cancelled**
  - Behavior: return to previous state; no error banner; no dirty-state change.
  - Threshold: immediate, no retry.
  - Fallback: none needed.

- **Unsupported file extension**
  - Behavior: show `Only .md, .markdown, and .mdown files are supported.`; keep previous document.
  - Threshold: reject before metadata/read calls.
  - Fallback: user can choose another file or use Save As with a supported extension.

- **CLI path invalid or unsupported**
  - Behavior: start in the empty state and show the mapped error banner.
  - Threshold: one startup attempt; no automatic retry and no automatic file dialog.
  - Fallback: user can use Open from the toolbar.

- **Multiple CLI paths provided**
  - Behavior: use the first path and show `Only one file can be opened at startup.`
  - Threshold: more than `1` positional file argument.
  - Fallback: user can open another file manually after startup.

- **File larger than `10 MB`**
  - Behavior: show `File is larger than 10 MB.`; keep previous document.
  - Threshold: metadata byte size greater than `10 * 1024 * 1024`.
  - Fallback: user can edit externally or split the file; app does not attempt streaming edit in v1.

- **Read failure or permission denied**
  - Behavior: show a mapped error; keep previous document; log original cause to console in development only.
  - Threshold: one read attempt per user action; no automatic retry.
  - Fallback: user can reopen via dialog, which may refresh runtime scope.

- **Save failure**
  - Behavior: show a mapped error; keep `dirty === true`; keep editor editable.
  - Threshold: one write attempt per user action; no automatic retry.
  - Fallback: user can use Save As to choose a new path.

- **Editor initialization failure**
  - Behavior: show source textarea mode automatically with an error banner saying WYSIWYG editor failed to load.
  - Threshold: if editor component throws during initialization or is not ready within `3 seconds`.
  - Fallback: source textarea remains fully editable and saveable.

- **Serialization failure**
  - Behavior: block save from WYSIWYG mode; switch to source mode with last known Markdown; show error.
  - Threshold: any exception while extracting Markdown from editor state.
  - Fallback: source mode save path writes `draftMarkdown` directly.

- **Dirty close confirmation unavailable**
  - Behavior: prevent internal Open/New transitions; for OS window close, use Tauri close event if available.
  - Threshold: if close event hook cannot be registered during startup, show persistent status warning.
  - Fallback: no autosave in v1; document that OS-level close protection is unavailable in that runtime.
